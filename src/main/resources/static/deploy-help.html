一键部署帮助手册

本文档详细说明“一键部署”功能的使用方式、Tomcat / TongWeb 两种中间件的部署流程，以及参数配置、YML 替换、心跳检测等关键逻辑，帮助运维人员在不同环境中稳定完成应用部署。

1. 产品概览

一键部署工具的目标是将“上传 WAR + 修改配置 + 停机重启”这一系列操作标准化、自动化：

  • 通过 Web 界面完成 参数配置、WAR 管理、YML 替换 等操作。
  • 支持 Tomcat 双实例（统一支撑 / 干部应用）和 TongWeb 两种中间件模式。
  • 部署过程自动完成：停止旧进程 → 拷贝 WAR → 解压 / 替换 YML → 修改端口 → 启动服务 → 心跳验证。
  • 支持生成 Linux 部署脚本，方便离线或批量环境中使用。

1.1 快速上手

标准的最小操作路径如下：

  • 步骤 1：在“一键部署”页选择 中间件类型、填写 部署目录、服务器 URL 与 数据库 IP / 账号密码。
  • 步骤 2：确认 wars/tyzc、wars/gbgl 下已经放置正确的 WAR 包，如有更新可上传 / 删除。
  • 步骤 3：为统一支撑、干部应用分别加载 YML 模板，点击“测试数据库连接”并保存 YML 配置。
  • 步骤 4：核对配置清单后点击“开始部署”，在右侧查看部署日志与自动的心跳检测结果。

2. 全局配置说明（Global Settings）

全局配置用于集中管理“一键部署”过程中的系统级默认值（如默认中间件类型、Tomcat 端口、JDK 路径、数据库连接串模板等），配置一次后可在多次部署中复用。

2.1 入口与保存位置

  • 配置入口：一键部署页面右上角的“全局设置”按钮，点击后会弹出全局设置弹窗。
  • 持久化位置：后端会将配置保存到 generated/settings/global-settings.json 文件，下次启动时自动加载。
  • 加载流程：页面加载时，前端调用 GET /api/settings/global，将返回结果写入内存中的 deployConfig.globalSettings，作为各步骤默认值来源。

2.2 关键字段说明

  • 默认中间件类型（defaultMiddlewareType）：
    • 当参数配置页未明确选择中间件类型时，后端会使用这里配置的默认值（默认 Tomcat）。
  • Tomcat 默认端口（tomcat.unifiedPort / tomcat.cadrePort / tomcat.authPort / tomcat.authUrlStripProtocol）：
    • 用于为端口配置提供初始值，以及构造 ${authurl}、心跳检测地址等。
    • authUrlStripProtocol 控制是否在写入 YML 时去掉 http(s):// 前缀，仅保留 host:port。
  • Tomcat 专用 JDK17 安装目录（tomcat.tomcatJdkHome）：
    • 当当前运行本工具的 JDK 版本低于 17 时，Tomcat 实例仍需在 JDK17+ 环境中启动。
    • 这里配置一个 JDK17 安装目录，例如 C:\Program Files\Java\jdk-17 或 /opt/jdk-17。
    • 参数配置第一步会调用 /api/deploy/java/check?jdkHome=... 验证此目录下的 bin/java 是否为 JDK17+，通过后会将路径写回该字段并在部署阶段用于设置 Tomcat 的 JAVA_HOME。
  • 数据库默认类型与达梦连接串模板（database.defaultType、database.dm.connectionPrefix、database.dm.connectionSuffix）：
    • 用于控制数据库默认类型（如“达梦”）以及连接串拼接规则（例如默认 jdbc:dm://）。
    • 测试数据库连接、YML 占位符替换时，都会使用这里的前缀 / 后缀来构造连接串。
  • YML 数据源类型（yml.datasourceTypeReplacement）：
    • 控制 YML 模板中 ${type} 占位符的替换值（默认 com.alibaba.druid.pool.DruidDataSource）。
    • 如需替换为其他连接池实现，只需在全局设置中修改该字段即可生效。

2.3 全局配置与部署流程的关系

  • 页面加载阶段：读取全局配置，为参数配置、端口配置、YML 模板等提供默认值。
  • 参数配置阶段：使用全局配置中的端口、数据库类型、JDK17 路径等信息进行校验；必要时引导用户修正配置并回写全局文件。
  • 开始部署阶段：直接消费已经校验通过的全局配置（包括 Tomcat JDK 路径、端口策略、YML 模板规则等），不再重复弹出 JDK 相关提示。

建议在首次使用前先打开“全局设置”，根据现场环境配置好中间件默认端口、数据库模板和 Tomcat 专用 JDK17 路径，可显著减少后续每次部署时的手工输入量。

3. 参数配置页说明步骤 1

参数配置页是整个部署流程的基础，所有后续 YML 替换和脚本生成都会依赖这里的配置。

3.1 基础参数

  • 中间件类型：Tomcat 或 TongWeb。
    • 选择 Tomcat 时，工具会根据预设端口（统一支撑 8111，干部应用 8222）自动修改 server.xml 并进行心跳检测。
    • 选择 TongWeb 时，工具会基于 domains/domain1/deployment 目录推导 TongWeb 安装目录和 bin/startd.sh / stopserver.sh。
  • 部署目录：
    • Tomcat：填写 Tomcat 安装根目录，例如 /usr/local/tomcat-tyzc。
    • TongWeb：填写 部署目录，例如 /usr/local/TongWeb8.x/domains/domain1/deployment。
  • 服务器 URL：
    • 格式示例：10.200.58.167:8111 或 http://10.200.58.167:8111。
    • 用于：
      • YML 中 ${authurl} 替换（干部应用也会引用统一支撑端口）。
      • 构造统一支撑 / 干部应用的心跳检测地址。

3.2 数据库配置

  • 数据库 IP：统一支撑与干部应用共用同一 IP，后端会分别组合生成两套连接串。
  • 统一支撑账号 / 密码：对应统一支撑库。
  • 干部应用账号 / 密码：对应干部库。
  • 点击“测试数据库连接”时，前端会：
    • 按 IP 拼接达梦连接串 jdbc:dm://IP 传给后端测试。
    • 测试成功后自动将连接信息写回当前 YML 文本中的占位符。

3.3 服务器 URL 与 authurl

  • Tomcat 模式：
    • ${authurl} 最终会被替换为 统一支撑 Tomcat 的 host:8111，例如 10.200.58.167:8111。
    • 干部分 YML 中的 ${authurl} 也会指向统一支撑端口，保证统一认证入口。
  • TongWeb 模式：
    • ${authurl} 直接使用你填写的服务器 URL 文本，不做前端协议补全。

4. Tomcat 部署中间件：Tomcat

4.1 目录与端口

  • Tomcat 安装目录：例如 /usr/local/tomcat-tyzc。
  • 应用 WAR 部署目录：$TOMCAT_HOME/webapps。
  • 默认端口策略：
    • 统一支撑 HTTP 端口：8111。
    • 干部应用 HTTP 端口：8222。
    • Shutdown / AJP 端口由工具自动推算，确保两个实例之间不冲突。

4.2 自动部署流程

点击“开始部署”后，Tomcat 模式的大致流程为：

  • 根据 HTTP / Shutdown 端口检测是否有旧实例在运行。
  • 优先调用 shutdown.sh 正常关闭，必要时按端口执行 kill -9 强制关闭。
  • 从 wars/tyzc、wars/gbgl 中复制相关 WAR 到对应 webapps 目录。
  • 对 tyzc-api.war、gbgl.war：
    • 解包到临时目录，替换 WEB-INF/classes/application-dev-dm.yml。
    • 重新打包为新的 WAR 再复制到目标目录。
  • 修改 server.xml 的 HTTP / HTTPS / AJP / Shutdown 端口。
  • 为 bin/*.sh 统一授予执行权限，调用 startup.sh 启动。

4.3 YML 替换逻辑

  • 统一支撑 / 干部应用分别有一套 YML 模板（步骤 3 中编辑）。
  • 数据库相关占位符在前端替换完成后，完整 YML 文本会写入 WAR 内部。
  • 升级或回滚时，只需更新外部 wars 目录中的 WAR，YML 替换逻辑保持不变。

4.4 心跳检测

  • 部署完成后，前端会监听 WebSocket 日志中“部署完成”关键字，随后开始心跳轮询。
  • 统一支撑心跳地址：http://<服务器IP>:8111/tyzc-api/heartbeat。
  • 干部应用心跳地址：http://<服务器IP>:8222/gbgl/heartbeat。
  • 实际 HTTP 请求由部署工具后端代理发起（/api/heartbeat/check），前端只展示结果，不受浏览器跨域限制。

5. TongWeb 部署中间件：TongWeb

5.1 目录结构要求

  • 典型部署目录：/usr/local/TongWeb8.x/domains/domain1/deployment。
  • 工具会从该路径向上回溯到 TongWeb 安装目录 TongWeb8.x，并推导出：
    • $TONGWEB_HOME/bin/startd.sh
    • $TONGWEB_HOME/bin/stopserver.sh
  • 部署前会通过 API 校验目录结构和脚本是否存在。

5.2 自动部署流程

  • 从 wars/tyzc、wars/gbgl 中复制 WAR 到 TongWeb 部署目录（扁平存放）。
  • 对 tyzc-api.war、gbgl.war 同样进行 YML 替换。
  • 为 startd.sh、stopserver.sh 授予执行权限。
  • 优先调用 stopserver.sh 关闭旧服务，再调用 startd.sh 启动新实例。

5.3 YML 替换逻辑

与 Tomcat 模式相同，只是最终 WAR 会被部署到 TongWeb 的部署目录中。

5.4 心跳与验证

  • 目前 TongWeb 模式的心跳 URL 仍以“服务器 URL + 路径”的方式构造，端口由服务器 URL 自行指定。
  • 同样通过后端代理接口进行检测，避免 CORS 问题。

6. WAR 管理步骤 2

  • 统一支撑 WAR 存放目录：wars/tyzc。
  • 干部应用 WAR 存放目录：wars/gbgl。
  • 页面支持：
    • 查看当前目录中的 WAR 列表。
    • 删除不需要的 WAR。
    • 上传新版 WAR（同名会覆盖）。

7. YML 配置页步骤 3

  • 统一支撑 / 干部应用各自维护一份 YML 文本。
  • 模板中包含占位符（如 ${type}、${url}、${authurl}），会在前端替换。
  • 高亮行帮助你快速定位被替换的关键配置段。

8. 部署脚本模式

“部署脚本”按钮会生成一份可在 Linux 上直接执行的 Shell 脚本：

  • 脚本包含：
    • 从本机 wars/ 目录复制 WAR 的逻辑。
    • 按需解包 / 替换 WAR 内的 application-dev-dm.yml。
    • Tomcat / TongWeb 的停止与启动命令。
  • 建议在脚本生成后，先在测试环境试跑确认无误，再推广到生产。

9. 数据初始化步骤 5

数据初始化功能用于在部署完成后执行初始化 SQL 脚本：

  • 可选择目标数据库（统一支撑 / 干部应用）。
  • 可指定脚本文件名列表（位于预定义的 resources/scripts/ 目录）。
  • 执行过程与结果会输出到“数据初始化”步骤下方的脚本日志区域。

10. 常见问题与排查

9.1 Tomcat 启动问题

  • 提示找不到或无法执行 startup.sh / catalina.sh：
    • 确认“部署目录”填写的是 Tomcat 安装根目录，而不是 bin 或 webapps。
    • 工具会自动对 bin/*.sh 执行 chmod +x，一般无需手动处理。
  • 端口占用：
    • 工具会尝试调用 shutdown.sh 及按端口 kill -9 清理旧进程。
    • 如仍被占用，需确认是否有其他服务使用了相同端口。

9.2 TongWeb 部署问题

  • 若提示未找到 startd.sh / stopserver.sh，请检查目录是否为 domains/.../deployment 结构。
  • 确认 TongWeb 安装完整且 bin 目录下存在对应脚本。

9.3 数据库与 YML 相关问题

  • 数据库连接测试失败时，根据提示检查网络、防火墙、账号密码和连接串格式。
  • YML 替换后应用仍无法连接数据库，请解包 WAR 检查内置 application-dev-dm.yml 是否为期望内容。

9.4 心跳检测异常

  • 如果直接在浏览器访问 heartbeat 正常，而工具显示失败，多半是端口或路径不一致，请对照日志检查构造的 URL。
  • 当前心跳检测由后端代理完成，不受浏览器跨域限制，但仍依赖统一支撑 / 干部应用自身的心跳接口可用。
